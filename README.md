# MydrecShop Telegram bot

Двуязычный магазин цифровых товаров, восстановленный по скриншотам старого
бота. Интерфейс работает на русском и английском, использует готовую анимацию
`assets/welcome.gif.mp4`, хранит каталог и заказы в SQLite и принимает ручную
оплату переводом USDT на Binance Pay ID.

## Возможности

- главное меню, каталог и карточки товаров в стиле старого бота;
- английский язык по умолчанию и русский язык с сохранением выбора пользователя;
- профиль, список заказов и подробный статус заказа;
- личный USDT-кошелёк каждого покупателя, пополнение через Binance Pay с
  ручным подтверждением администратора и оплата заказов с баланса;
- уникальный шестизначный `note` заказа, приём Binance Transaction ID и ручная
  проверка платежа администратором;
- отдельная очередь «Платежи на проверке», куда заказ попадает сразу после
  нажатия «Я оплатил», даже если Transaction ID ещё не введён;
- атомарный резерв конкретных аккаунтов на 10 минут: покупатель может отменить
  неоплаченный заказ кнопкой «Отменить заказ», а отклонение администратором
  возвращает аккаунты в наличие;
- отдельная база аккаунтов для каждого товара и защита от повторной выдачи;
- отдельная кнопка `⚙️ Настройки бота` только для администраторов;
- создание и редактирование товаров, цены, описания, видимости и
  склада аккаунтов через кнопки админ-панели;
- автоматический перевод названия, описания и гарантии товара между русским и
  английским при создании и редактировании;
- уведомление пользователей о пополнении на выбранном ими языке;
- массовая рассылка текста, фотографии, видео или файла с предпросмотром и
  подтверждением администратора;
- обязательная подписка на заданный Telegram-канал с повторной проверкой;
- выбор количества при покупке через `−`/`+`, быстрые кнопки `5`/`10`/`15`
  или ручной ввод и автоматическая выдача ровно оплаченного числа аккаунтов;
- premium/custom emoji в карточках и inline-кнопках через `tg-emoji` и
  `icon_custom_emoji_id`;
- автоматический fallback на обычные Unicode-эмодзи;
- SQLite без отдельного сервера, атомарное резервирование последней единицы.

## Быстрый запуск в Windows PowerShell

1. Создайте нового бота через `@BotFather` и сохраните токен.
2. В корне проекта выполните:

```powershell
py -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -e ".[dev]"
Copy-Item .env.example .env
```

3. Откройте `.env` и заполните как минимум:

```dotenv
BOT_TOKEN=123456:replace_me
ADMIN_IDS=123456789
BINANCE_ID=123456789
REQUIRED_CHANNEL_USERNAME=mydrecsales
SUPPORT_USERNAME=AWP_ON
PAYMENTS_ENABLED=false
```

Числовой Telegram ID можно узнать у `@userinfobot`. Реальный `.env` уже
добавлен в `.gitignore`; не отправляйте и не коммитьте токен.

4. Запустите:

```powershell
python -m mydrecshop
```

При первом старте бот создаст `data/shop.db` и наполнит каталог товарами со
скриншотов. Начальный остаток товаров равен нулю: продажа становится доступна
только после загрузки реальных строк аккаунтов через `⚙️ Настройки бота`. До публикации
проверьте цены USDT, описания и условия продажи каждого товара. Переменная
`PAYMENTS_ENABLED` задаёт состояние продаж только при первом создании базы.
После этого продажи включаются и выключаются прямо в админ-панели кнопками
`🟢 Включить продажи` и `🔴 Выключить продажи`, без перезапуска и деплоя.
Состояние хранится в SQLite и переживает обновления сервиса.

Кнопки `🛠 Включить техработы` и `✅ Завершить техработы` управляют отдельным
режимом технического обслуживания. Пока он включён, обычные пользователи
получают локализованное уведомление вместо выполнения команд и нажатий, а все
администраторы из `ADMIN_IDS` продолжают работать с ботом. Переключатель продаж
при этом не изменяется, фоновые таймеры резервов продолжают работать, и после
завершения техработ магазин возвращается к прежнему состоянию.

У каждого товара в управлении есть кнопка `📊 Настроить оптовые цены`.
Пороги вводятся одним сообщением (`5=0.90`, `10=0.85`, `15=0.80`), а `off`
удаляет все оптовые пороги. Бот показывает сетку покупателю и атомарно фиксирует
подходящую цену при резервировании заказа.

Отдельно проверьте, разрешена ли перепродажа каждого типа аккаунта правилами
его поставщика и законодательством вашей страны. Проект технически реализует
витрину и оплату, но не подтверждает право перепродажи сторонних сервисов.

## Premium/custom emoji

Telegram теперь позволяет задать одну premium-иконку перед текстом inline-
кнопки. При создании товара или изменении его названия достаточно отправить
название с premium emoji в начале: бот отделит Telegram ID от Unicode fallback,
сохранит его отдельно и автоматически покажет иконку в каталоге. Команда
`/setemoji` остаётся запасным способом — отправьте её вместе с emoji или
ответьте командой на сообщение с emoji:

```text
/setemoji hotmail-outlook-mail
```

ID иконок главного меню можно задать в `.env` (`EMOJI_CATALOG_ID`,
`EMOJI_ORDERS_ID` и другие). Для использования custom emoji владелец бота
должен иметь Telegram Premium либо бот должен владеть дополнительным username
с Fragment. При отказе API клавиатура автоматически повторяется с Unicode-
иконками. `/setemoji` также сохраняет обычный emoji из того же стикера, поэтому
корректный fallback появляется и в карточке товара без ручного поиска ID.

## Администрирование

Пользователь из `ADMIN_IDS` получает в главном меню личную кнопку
`⚙️ Настройки бота`. Обычным пользователям она не показывается. В панели можно:

- добавлять товары с полным названием и эмодзи;
- менять название, описание, цену USDT и видимость товара;
- загружать и списывать конкретные аккаунты, при этом stock меняется автоматически;
- открывать оплаченные заказы и выдавать ровно купленное количество аккаунтов;
- создавать рассылку всем пользователям с предпросмотром перед отправкой;
- скачивать защищённую резервную копию базы.
- просматривать заявки на пополнение баланса и подтверждать либо отклонять их;
- пополнять, списывать и проверять баланс клиента служебными командами.

Служебные команды также доступны только ID из `ADMIN_IDS`:

```text
/admin                     кнопочная админ-панель
/products                  список товаров и их SKU
/backup                    отправить админу online backup SQLite
/balance <telegram_id>     показать баланс клиента
/addbalance <id> <USDT> [примечание]
                            пополнить баланс клиента
/subbalance <id> <USDT> [примечание]
                            списать средства с баланса клиента
/setprice <sku> <USDT>     установить цену в USDT
/setemoji <sku>            сохранить custom emoji из сообщения/ответа
/hide <sku>                скрыть товар
/show <sku>                показать товар
/deliver <order_id> <data> аварийная ручная выдача
/cancelorder <order_id>    отменить неоплаченный заказ
/reconcile_delivery <order_id>
                            записать уже выполненную выдачу без повторной отправки
/reconcile_refund <order_id> <charge_id>
                            записать уже выполненный Telegram-возврат без его повтора
```

Ту же панель можно открыть командой `/admin`.
При пополнении вставьте по одному аккаунту на строку либо отправьте UTF-8 `.txt`.
Дубликаты пропускаются, а stock увеличивается только на фактически добавленные
строки. При создании заказа stock и конкретные строки резервируются атомарно. Если за 10 минут
покупатель не нажал «Я оплатил», резерв возвращается автоматически. После «Я оплатил» резерв снимается
только при отклонении заказа администратором, а
после подтверждения платежа кнопка «Отправить аккаунты» выдаёт покупателю ровно
`quantity` строк из базы выбранного товара.
Если уведомление об оплате потерялось, все подтверждённые заказы доступны через
`⚙️ Настройки бота` → «Заказы к выдаче».

Кнопка `💰 Кошелёк` доступна в главном меню. Покупатель вводит произвольную
положительную сумму USDT, получает Binance ID и уникальный `BAL-...` Note,
после перевода нажимает «Я отправил» и присылает Transaction ID. Деньги
появляются на внутреннем балансе только после ручного подтверждения администратором.
Повторный клик не зачисляет сумму второй раз, а один Transaction ID нельзя
использовать одновременно для заказа и пополнения кошелька.

При вводе русского текста товара бот создаёт английскую версию, при вводе
английского — русскую. Для перевода используется внешний сервис MyMemory;
если он временно недоступен или исчерпан дневной лимит, исходный текст
сохраняется в обеих версиях, а администратор получает предупреждение.

Аккаунты содержатся в SQLite в исходном виде, поэтому защищайте `.env`, файл
`data/shop.db` и резервные копии. Не публикуйте базу и не пересылайте backup в
незащищённые чаты.

Команды `reconcile_*` аварийные. Используйте их только после ручной проверки,
что товар действительно уже отправлен.

## Обязательная подписка на канал

Укажите публичный username канала без `@` в `REQUIRED_CHANNEL_USERNAME`.
Добавьте бота администратором этого канала; права на публикацию и удаление
сообщений ему не требуются. Пока бот не является администратором, проверка
безопасно остаётся неактивной и администратору бота приходит предупреждение.
После настройки неподписанный пользователь видит кнопки перехода в канал и
повторной проверки вместо функций магазина. Администраторы из `ADMIN_IDS`
проходят проверку без подписки.

## Загрузка на Render

Проект уже содержит `Dockerfile` и `render.yaml`. Для long polling + SQLite
нужен **Background Worker Starter** и постоянный диск. Бесплатного Worker у
Render нет; бесплатный Web Service засыпает без входящих HTTP-запросов и не
даёт подключить диск, поэтому для такого бота он ненадёжен. На июль 2026 года
минимальная конфигурация стоит примерно $7 за Worker + $0.25 за диск 1 GB в
месяц.

### 1. Подготовьте репозиторий

Создайте приватный репозиторий GitHub и загрузите туда весь проект, кроме
`.env`, `.venv` и `data/shop.db` (они уже исключены через `.gitignore`).
Перед первым деплоем подготовьте зашифрованный согласованный снимок текущей БД:

```powershell
python scripts/prepare_deploy_snapshot.py
python scripts/verify_deploy_snapshot.py
```

Файл `deploy/bootstrap/shop.db.fernet` можно коммитить: он зашифрован и
аутентифицирован. Ключ остаётся только в игнорируемом файле
`data/deploy-bootstrap.key`; не публикуйте и не отправляйте его в репозиторий.
Из корня проекта это можно сделать так (подставьте URL своего пустого
репозитория):

```powershell
git add .
git commit -m "Initial MydrecShop bot"
git branch -M main
git remote add origin https://github.com/YOUR_NAME/YOUR_REPOSITORY.git
git push -u origin main
```

Если `origin` уже существует, вместо `git remote add` используйте
`git remote set-url origin ...`. Перед `git push` выполните `git status` и
убедитесь, что `.env` и `data/shop.db` не попали в список файлов.

Обязательно проверьте, что в репозитории есть:

```text
Dockerfile
render.yaml
pyproject.toml
src/
assets/welcome.gif.mp4
deploy/bootstrap/shop.db.fernet
```

### 2. Создайте Blueprint

1. В Render нажмите **New → Blueprint**.
2. Подключите GitHub-репозиторий с ботом.
3. Render прочитает `render.yaml` и предложит создать
   `mydrecshop-bot` (Background Worker Starter) и диск 1 GB.
4. Введите секреты, которые Render запросит:
   - `BOT_TOKEN` — токен нового бота от `@BotFather`;
   - `ADMIN_IDS` — ваш числовой Telegram ID; несколько ID через запятую;
   - `BINANCE_ID` — числовой Binance Pay ID, на который покупатели переводят USDT;
   - `DATABASE_BOOTSTRAP_KEY` — содержимое локального
     `data/deploy-bootstrap.key`;
   - `DATABASE_BOOTSTRAP_ENABLED` — `true` только для первого запуска нового диска.
5. После создания откройте **Environment** и при необходимости добавьте
   `SUPPORT_USERNAME`. Если переменная отсутствует, используется `AWP_ON`.
   Там же можно добавить `EMOJI_CATALOG_ID`, `EMOJI_ORDERS_ID`,
   `EMOJI_PROFILE_ID`, `EMOJI_LANGUAGE_ID`, `EMOJI_BUY_ID` и `EMOJI_BACK_ID`.
6. Подтвердите создание сервисов и дождитесь сборки Docker image.

### 3. Проверьте запуск

Откройте вкладку **Logs**. Успешный запуск заканчивается строкой вида:

```text
Starting @your_bot with long polling
```

После этого отправьте боту `/start`, переключите язык и проверьте каталог.
После успешного восстановления БД и повторного рестарта обязательно установите
`DATABASE_BOOTSTRAP_ENABLED=false`: старый снимок больше не должен применяться
к новому пустому диску. `PAYMENTS_ENABLED` служит только начальным значением;
дальше используйте переключатель продаж в админ-панели. До включения проверьте
Binance ID, цены и содержимое зашифрованного снимка. Не
запускайте одновременно локальную копию с тем же токеном: два процесса
`getUpdates` будут конфликтовать.

SQLite хранится на диске `/app/data`, поэтому каталог, пользователи и заказы
переживают рестарты и новые деплои. Анимация входит в Docker image и отдельного
диска не требует. Blueprint фиксирует `numInstances: 1`, поскольку SQLite и
long polling не должны обслуживаться несколькими экземплярами одновременно.

### 4. Обновления

После `git push` Render автоматически пересоберёт и перезапустит Worker.
Постоянный диск не удаляется. При изменении секретов обновляйте их через
**Environment**, а не добавляйте `.env` в Git.

Render делает снимки диска, но они не заменяют резервную копию самой базы.
Регулярно выполняйте `/backup`. При наличии `BACKUP_ENCRYPTION_KEY` (или
`DATABASE_BOOTSTRAP_KEY`) бот отправляет файл `.sqlite3.fernet`; без ключа он
явно предупреждает, что backup не зашифрован. Команда использует online backup
SQLite, поэтому её можно запускать без остановки бота.
